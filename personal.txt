მუშაობა დასრულდა

აქ მოცემულია შენიშვნები/შემოწმებები, რომლებსაც ყოველთვის გამოვიყენებ, როცა მობილურ აპლიკაციას ბექენდთან დააკავშირებ:

"ზუსტად რომელ URL-ს უნდა მიაღწიოს აპლიკაციამ Django-სთვის?"

დაადასტურეთ ბექენდურლი (http://<LAN-IP>:8000/, ngrok URL, ან წარმოების ჰოსტი) და დარწმუნდით, რომ როგორც Expo Go-ს, ასევე მოწყობილობას შეუძლია მასთან წვდომა.

"შეესაბამება თუ არა ჩემი Expo კონფიგურაცია (app.json → extra.backendUrl/realtimeUrl) იმ გარემოს, რომელსაც ვამოწმებ?"

ორჯერ შეამოწმეთ ეს მნიშვნელობები, როცა ქსელს ან ბექენდს ცვლით.

"Django-ს ვუშვებ python manage.py runserver 0.0.0.0-ით (ხაზი 8000) (ან ngrok-ის უკან), რათა სხვა მოწყობილობებმა შეძლონ მასთან წვდომა?"

დაკავშირებული პორტები/ჰოსტები მნიშვნელოვანია; 127.0.0.1 მხოლოდ იმავე მოწყობილობაზეა ხელმისაწვდომი.

"შესვლის შემდეგ, პასუხში ვხედავ თუ არა მოქმედ ტოკენს და ვაახლებ თუ არა ტოკენს?"

კლიენტი ორივეს ელის; დარწმუნდით, რომ ბექენდი აბრუნებს მათ და აპლიკაცია ინახავს მათ

"შეესაბამება თუ არა ჩემი API პასუხები src/services/api/*-ში კოდირებულ კონტრაქტებს (ველის სახელები, მონაცემთა ტიპები)?"

თუ backend სქემა შეიცვლება, განაახლეთ TypeScript ტიპები + UI შესაბამისობისთვის.

"რეალურ დროში აკრეფის/მოვლენებისთვის: არის თუ არა WebSocket-ის კარიბჭე (realtimeUrl) ხელმისაწვდომი მოწყობილობიდან?"

გადაამოწმეთ TLS/hostnames/firewalls, წინააღმდეგ შემთხვევაში აპლიკაცია კვლავ გამოკითხვას დაიწყებს.

"Django-ს CORS/CSRF პარამეტრები იძლევა თუ არა Expo ჰოსტიდან მოთხოვნებს?"

საჭიროების შემთხვევაში, დაშვებულ წყაროებში დაამატეთ http://localhost:8081, exp://..., ან თქვენი მოწყობილობის IP.

"მჭირდება თუ არა რაიმე მონაცემის (გეგმების, მომხმარებლების) დათესვა ფუნქციის ტესტირებამდე?"

გადახდები, მენტორები და ა.შ. დამოკიდებულია იმაზე, რომ backend-ს აქვს მონაცემები; ცარიელი მონაცემთა ბაზა დააბრუნებს ცარიელ ბარათებს.

გაითვალისწინეთ ეს შენიშვნები ყოველ ჯერზე, როდესაც ახალ გარემოს ჩართავთ და თქვენ აღმოაჩენთ კავშირის პრობლემების უმეტესობას, სანამ ისინი შეგექმნებათ

Finished working

Here are the prompts/checks I'd keep handy whenever you hook the mobile app to a backend:

"What exact URL should the app hit for Django?"

Confirm the backendurl (http://<LAN-IP>:8000/, ngrok URL, or production host) and make sure both Expo Go and the device can reach it.

"Does my Expo config (app.json → extra.backendUrl/realtimeUrl) match the environment I'm testing?"

Double-check those values whenever you switch networks or backends.

"Am I running Django with python manage.py runserver 0.0.0.0 (line 8000) (or behind ngrok) so other devices can reach it?"

Bound ports/hosts matter; 127.0.0.1 is only accessible on the same machine.

"After logging in, do I see valid token and refresh Token in the response?"

The client expects both; ensure the backend returns them and the app stores them.

"Do my API responses match the contracts coded into src/services/api/* (field names, data types)?"

If the backend schema changes, update the TypeScript types + UI to match.

"For realtime typing/events: is the WebSocket gateway (realtimeUrl) accessible from the device?"

Verify TLS/hostnames/firewalls, otherwise the app falls back to polling.

"Are CORS/CSRF settings on Django allowing requests from the Expo host?"

Add http://localhost:8081, exp://..., or your device IP to allowed origins if needed.

"Do I need to seed any data (plans, users) before testing a feature?"

Payments, mentors, etc., rely on the backend having data; a blank DB will return empty cards.

Keep these prompts in mind each time you spin up a new environment, and you'll catch most connectivity issues before they bite.
